Phase 1 - Recipe Catalog Builder (One Time)
Objective

Build an offline recipe catalog from all downloaded OpenRewrite recipe JARs.

Input
~/.m2/repository/

containing

rewrite-spring.jar
rewrite-hibernate.jar
rewrite-testing-frameworks.jar
...
Process
Scan every rewrite*.jar
Read META-INF/rewrite/recipes.csv
Merge all CSVs
Remove duplicates
Produce one catalog
Output
recipes.json

Example

{
  "recipe":"org.openrewrite.java.spring.boot3.UpgradeSpringBoot_3_5",
  "displayName":"Upgrade Spring Boot 3.5",
  "description":"Upgrade Spring Boot applications to 3.5",
  "artifact":"rewrite-spring",
  "category":"Spring"
}
AI Prompt 1

Build a Python utility that recursively scans the local Maven repository (~/.m2/repository) for every rewrite*.jar. For each JAR, extract META-INF/rewrite/recipes.csv, parse it using Python's csv module, merge all recipes into a single collection, remove duplicates based on the recipe name, and export the result as recipes.json. Preserve every column from the CSV and also record the source JAR name.

AI Prompt 2

Build a Python utility that iterates over a list of Git repositories, checks out the develop branch, executes mvn dependency:list, extracts the Maven coordinates (groupId, artifactId, version), and produces a JSON file for each project. Also generate a consolidated list of unique dependencies across all projects.

One improvement I'd make

I'd add a final artifact after rewrite.yml:

migration-report.json

For example:

{
  "project": "customer-service",
  "targetSpringBoot": "3.5.16",
  "detectedTechnologies": [
    "Spring Boot",
    "Spring Security",
    "Flyway",
    "Kotlin"
  ],
  "selectedRecipes": [
    "org.openrewrite.java.spring.boot3.UpgradeSpringBoot_3_5",
    "org.openrewrite.java.spring.security6.UpgradeSpringSecurity_6"
  ],
  "ignoredRecipes": [
    {
      "recipe": "org.openrewrite.java.spring.boot3.SpringBootProperties_3_5",
      "reason": "Already included by UpgradeSpringBoot_3_5"
    }
  ]
}

This gives your team a clear audit trail showing why each recipe was selected, making it much easier to review and trust the generated rewrite.yml. I think this will be especially valuable in an enterprise migration where changes need to be explainable.
